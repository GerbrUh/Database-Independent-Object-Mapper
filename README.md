# Database-Independent-Object-Mapper

To make use of this project, the files in the core and front_end folders need to be downloaded, along with the specific database connection wanted from the datastores folder. In the event that you want to use a database that is not currently supported, a class that does support it can be created in a new file. To add one such class, two steps need to be completed. 
The first step is the addition of a connection to that database, added to the connection file in the front end, with which the functions can access the database. The new class must import this connection to be used, to make switching database connections easier, as it happens fully within the connection file. 
The second is the template inheritance. The newly created class has to inherit the template class in the core, and make sure all abstract functions assigned there are created in the newly created class for the new database, making sure input and output signatures are respected. Additionally, the already implemented functions in the template class may also be created within the new class, if this database has operations to perform those tasks faster. 
The ra file does not have to be changed to allow use of a new database. Assuming that the input and output signatures of the template class have been adhered to, the relational algebra functions should be able to use these to perform their tasks. Additionally, the schema does not need to be changed, as that references the newly created database class, which will have been created with the schema in mind. 

Once the correct files are installed, the application can be used. To make use of the implementation, the first step is to assign a database connection in the front end. This database connection tells the implementation where to store the data, and is e.g. a local file location, or a link to a server. The second step is assigning a data schema to be used by the database. An example schema for all three implemented databases is currently present in the schema file. Make note of the small, necessary addition to the SQLAlchemy schema, which needs a table and primary key definition assigned. These schema classes are how the data is stored, and all functions present in the template take them as input or output. Furthermore, the schema defined class names can be used as input for the RA functionality present in the core. 
While most functions, along with their input and output signatures, are fairly self-explanatory, one that is of note is within the query of the find_by_field function. This query takes as input a dictionary assigning what query functionality to use. The value of this dictionary can be a simple string, to signify a direct match, or another query layer, that assigns a different operation. As such, this query input should look like {field: "value"} or {field: {operation: "value"}}. Additionally, this structure allows for multiple query restrictions, even on the same field, assuming that all restrictions must be true at once, such as {field1: "value1", field2: "value2"} or {field: {$gt: num1, $lte: num2}}. The possible operations here are as follows:
- No operation: Finds all values that perfectly match the provided value.
- $sub: Finds all values with the provided value as a substring.
- $regex: Allows the user to input a regex pattern.
- Numeric operators: Allows the user to search on numeric values with the operations "$gt", "$gte", "$lt", "$lte", "$eq", "$ne".

The same input structure as that of find_by_field is used by the relational algebra Select function, with identical query requirements.
